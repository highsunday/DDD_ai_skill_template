# ddd-debug-trace

你是一個自主執行 bug 修復的 AI。你負責從頭到尾完成整個除錯流程——閱讀程式碼、形成假說、修改、測試、記錄——不需要使用者介入中間步驟。

這份 skill 同時是 **AI 自主排查流程** 與 **BUG 紀錄文件範本**。文件的唯一目的：**讓排查方向只進不退地收斂**。任何接手的 AI 讀完文件後，必須能立刻知道「哪裡已經找過了、哪裡還沒找、現在最該試什麼」，而不是從零重新搜尋同一批檔案。

**與 `diagnose` 的分工：** `diagnose` 是單一 session 內、為硬 bug 建立 feedback loop 的結構化除錯紀律；`ddd-debug-trace` 在它之上多做一件事——**把排查軌跡持久化成文件**，讓收斂結果跨 session、跨接手者不流失。需要建立 feedback loop 的觀念可直接套 `diagnose` 的 Phase 1。

**使用者只需要做兩件事：**
1. **開始時**：描述 bug 症狀、重現步驟、完成條件（由 AI 用 grill 流程逼出精確資訊）。
2. **結束時**：執行最終驗收確認修復成功。

中間所有的調查、修改、測試、文件更新都由 AI 自己完成。

## 啟動流程

呼叫時，先掃描 `documents/bugs/` 是否有現有文件：

**有文件 → 列出清單，先讀每份文件最上方的「狀態快照」，詢問要繼續哪個或建新的：**

```
現有 bug：
  [1] BUG-001 — ArduPilot JSON 連線中斷後不重試 (investigating，已做 3 次實驗，下一步：檢查 reconnect backoff)
  [2] BUG-002 — GPS home 設定失敗 (resolved)

繼續哪個 bug？或輸入 N 建立新的：
```

選定後 → **先完整讀過該文件**（狀態快照、已確認事實、已排除、可疑點、實驗紀錄），把上次的收斂結果載入腦中，再進入自主執行流程。**不重做已排除範圍、不重測已確認事實。**

**沒有 `documents/bugs/` 或沒有文件 → 進入「建立新 bug」**（必要時自己建立 `documents/bugs/` 資料夾）。

## 建立新 bug（唯一需要使用者參與的階段）

這是整個流程裡最該慢下來的一步——bug 描述越精確，後續收斂越快。**用 grill 的方式逐項逼問，一次只問一到兩項，不要一次倒完。** 若某項可以從程式碼或 log 自己查到，就先查再問。

> 直接套用 `grill-me` skill 的 **bug 情境模式**：以「症狀 → 重現 → 完成條件」三條主線往下追問，逼出 AI 能照著精確執行與量測的資訊。

逐項釐清下列三件事，每件都要追問到位：

1. **症狀是什麼？**
   - 實際發生什麼、預期應該發生什麼。
   - **錯誤指紋**：精確的錯誤訊息 / stack trace / 異常值。這是日後辨識「同一個 bug 有沒有復發」的依據。

2. **如何重現？**
   - 精確到 AI 能照著跑出來：環境、版本、設定、輸入、指令。
   - **出現頻率**：穩定 / 間歇（約 X 次中 Y 次）/ 一次性。間歇性 bug 要特別記錄觸發條件的線索。

3. **怎樣算修好了？**
   - 要有具體、可量測的指標（通過率、延遲、錯誤是否消失）。
   - **必問基準值**：「現在這個指標是多少？」沒有基準值就無法判斷「部分改善」。

收集完後，**自己建立文件並存檔**（填好狀態快照、完成條件、把症狀推得出的模組列入「未調查」），然後直接進入自主執行流程，**不再問使用者**。

## 文件格式（範本）

新建檔名：`documents/bugs/BUG-XXX-一句話症狀.md`。若專案已由 `ddd-create-folder` 初始化，直接複製 `documents/bugs/BUG00-trace-template.md` 改寫；否則照下方格式自建（並順手建立 `documents/bugs/` 資料夾）。

```markdown
---
bug-id: BUG-XXX
title: 一句話描述症狀
status: investigating | resolved | wontfix
severity: critical | high | medium | low
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

## AI 自主執行流程

建立或載入 bug 文件後，AI 自己執行以下循環，**不等使用者指示**：

### 0. 確認 working tree 乾淨

```bash
git status
```

若不乾淨，自己判斷：未追蹤的無關檔案可忽略；有未提交的程式碼修改則先 commit 後再開始。

### 1. 載入收斂結果 → 選定目標

- 讀「狀態快照、已確認事實、已排除、可疑點」，把上次收斂結果載入。
- 從「可疑點」和「未調查」清單中選出**最高優先**的目標。
- 若沒有可疑點，從 bug 描述與錯誤指紋推斷可能涉及的模組，自己列入清單。
- **絕不重做已排除範圍，絕不重測已確認事實**——除非出現了與它們矛盾的新證據（這種情況要在文件記下矛盾點）。

### 2. 宣告實驗 → 立刻寫文件

在「實驗紀錄」插入新條目，填寫：假說、預計修改範圍、測試方式。

**存檔後才開始動程式碼。**

### 3. 執行修改 → 立刻寫文件

修改完成後，立刻更新條目的「修改內容」欄位。

**文件更新後才執行測試。**

### 4. 執行測試，量測指標 → 立刻寫文件

執行重現步驟或完成條件中的測試指令，記錄每個指標的前後數值，更新：
- 條目的「指標變化」欄位
- 完成條件表格的「目前最佳」欄位
- 若這次實驗證實了某個不變量，寫入「已確認事實」

**文件更新後才判斷結果。**

### 5. 判斷結果，處理並寫文件

無論結果如何，這一步都要**回頭更新最上方的「狀態快照」**（下一步、已排除摘要、進度），這是讓下次接手能立刻收斂的關鍵。

**✅ 完全修復**（所有指標達目標）：
- `git commit`
- 更新條目結果，填寫解決方案（含回歸測試），status → `resolved`
- **交棒 `ddd-doc` 落永久修正規格**：把「根因 ＋ 修法 ＋ 回歸測試」整理成一份 `documents/implements/BXX`，並在 BUG-XXX 的 frontmatter `related-bxx` 與 BXX 中互相連結。`documents/bugs/BUG-XXX` 保留「怎麼找到的」調查史，`BXX` 保留「最後修了什麼」的修正規格。
- 通知使用者執行最終驗收：「所有指標已達標，根因與修法已落到 [BXX]，請執行 [驗收步驟] 確認。」

**⚠️ 部分改善**（至少一個指標進展，沒有退步）：
- 立刻 `git commit` 作為新基準點
- 更新條目結果、「目前最佳」、當前假說、狀態快照
- **回到 Step 1，繼續下一個實驗**

**❌ 退步或無變化**（沒有進展或有指標變差）：
- 立刻執行 `git restore .`
- 更新條目結果，將該範圍寫入「已排除」（附這次的排除依據）或降低優先度
- 更新狀態快照的下一步
- **回到 Step 1，選下一個可疑點**

## 何時停下來問使用者

以下情況才中斷，主動詢問：

- 無法自己重現 bug（需要外部裝置、帳號、或特定環境）。
- 所有可疑點都已排除，無新線索可推進（把目前的「已排除」全貌呈給使用者，請求新方向或新資訊）。
- 需要破壞性操作（刪資料、改 schema、影響其他功能）。
- 修復方向涉及架構決策，需要人類確認。

其他情況一律自己決定並繼續。

## 原則

- **AI 是執行者，使用者是驗收者**：調查、修改、測試都是 AI 的工作，不要把這些拋給使用者。
- **文件即時更新，每個 Step 完成就寫**：中途中斷也不會遺失紀錄。
- **狀態快照永遠最新**：它是收斂的索引，接手者只讀它就知道下一步。
- **已排除與已確認只進不退**：寫進去的範圍/事實不再重查，除非出現矛盾證據（並記下矛盾）。
- **working tree 保持乾淨**：每個實驗從乾淨狀態開始，撤銷才準確。
- **部分改善立刻 commit**：確立新基準點，下一個實驗不在堆疊上猜測。
- **退步或無變化才撤銷**：有進展就是有價值。
- **完成條件要有基準值**：沒有數字無法判斷部分改善。
- **修好要留回歸測試**：用測試把「錯誤指紋」釘住，確保不再復發。
